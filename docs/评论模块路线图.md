# 评论模块路线图

> 目标：为博客文章页增加轻量、稳定、低维护的评论能力，同时不破坏当前博客的极简阅读体验。

## 一、设计目标

评论模块要满足四个原则：

- **静态站点友好**：不能强依赖自建后端
- **维护成本低**：尽量避免数据库、审核后台、登录系统的额外负担
- **垃圾评论可控**：优先利用现成身份体系过滤低质量内容
- **视觉克制**：评论区是正文附录，不是喧宾夺主的主角

## 二、候选方案对比

### 方案 A：Giscus（推荐）

基于 GitHub Discussions。

优点：
- 不需要自建服务端
- 不需要数据库
- GitHub 登录天然过滤大量垃圾评论
- 非常适合技术博客读者
- 与 Astro 集成简单
- 后续可迁移，锁定成本较低

缺点：
- 读者需要 GitHub 账号
- 评论生态更偏技术用户
- 交互体验受 GitHub Discussions 能力边界限制

结论：

> 对当前 clawasync 博客阶段，Giscus 是最均衡、最务实的选择。

---

### 方案 B：Twikoo / Waline

优点：
- 可支持匿名或更宽松的评论方式
- 可扩展通知、后台管理等能力
- 用户门槛更低

缺点：
- 需要部署服务端或云函数
- 要面对垃圾评论、审核、维护问题
- 整体复杂度和长期维护成本明显更高

结论：

> 如果未来博客从“技术写作阵地”升级为“大众内容站点”，再考虑这一类方案更合适。

---

### 方案 C：先不上真实评论，只做反馈入口

形式包括：
- 跳转 GitHub Discussion
- 邮件反馈
- 外部表单

优点：
- 极轻
- 几乎零维护

缺点：
- 互动感弱
- 不形成文章页内讨论沉淀

结论：

> 适合作为过渡方案，但不适合作为长期主方案。

## 三、推荐决策

当前推荐顺序：

1. **先接入 Giscus**
2. 观察真实互动量与评论质量
3. 再判断是否有必要升级为更重的评论系统

一句话判断：

> **先用 GitHub 身份做高质量评论过滤，再决定要不要做大众化评论。**

## 四、产品形态建议

### 评论区在页面中的位置
建议放在：

- 正文结束后
- 作者信息或文章结尾区之后
- 相关推荐或下一篇导航之前

不建议：
- 插在正文中间
- 放到首屏可见区域
- 设计成过于显眼的大卡片

### 视觉风格建议
目标是延续当前博客的极简、克制气质：

- 评论区标题弱化处理
- 外边距充足，和正文有明显分隔
- 不使用高饱和强调色
- 不使用厚重阴影和强卡片感
- 评论容器尽量清爽，像附录讨论区

## 五、实施路线图

### 阶段 1：方案固化
目标：确定评论系统选型与边界。

任务：
- 确认采用 Giscus
- 确认使用哪个 GitHub 仓库承载 Discussions
- 确认 Discussion 分类
- 确认文章与评论线程的映射策略

建议：
- 映射优先使用 `pathname` 或稳定 `slug`
- 不建议用标题，避免后续文章改名造成映射漂移

### 阶段 2：组件接入
目标：把评论模块挂到文章页，但不破坏性能和排版。

任务：
- 新建评论组件
- 在 `src/pages/post/[slug].astro` 中接入
- 保证仅文章页加载，不污染其他页面
- 确保移动端可用

### 阶段 3：样式收敛
目标：让评论区融入博客整体气质。

任务：
- 调整评论区上下间距
- 统一标题、边框、留白风格
- 降低默认嵌入样式的突兀感
- 检查暗色/浅色兼容策略（若后续支持主题切换）

### 阶段 4：体验校验
目标：验证评论系统是否真正值得长期保留。

任务：
- 检查加载性能
- 检查首屏影响
- 验证 Discussion 自动创建与映射是否稳定
- 实际观察一段时间的评论质量和互动频次

## 六、工程建议

建议涉及文件：

- `src/pages/post/[slug].astro`
- `src/components/CommentSection.astro` 或类似组件
- `src/styles/global.css`

建议把评论逻辑封装成单独组件，不要把第三方嵌入代码直接散落在文章模板里。这样后续切换评论提供方时会轻松很多。

## 七、验收标准

评论模块完成后，至少要满足：

- 仅文章页显示
- 不依赖自建后端
- 评论线程映射稳定
- 页面整体风格不被破坏
- 首屏性能影响可接受
- 垃圾评论风险可控

## 八、暂不做的事情

在第一版里不建议一起做：

- 自建评论数据库
- 匿名评论
- 邮件订阅联动
- 点赞、表情、打赏等重互动模块
- 复杂后台审核系统

先把“稳定可用的讨论区”做出来，比一开始做全套互动系统更重要。
