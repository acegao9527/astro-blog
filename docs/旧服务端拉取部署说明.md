# 旧服务端拉取部署说明

这条服务端拉取部署链路已退出常规发布流程。当前常规发布由 blog 仓库的 `blog-website-publish` skill 驱动：

```text
blog-website-publish skill
  -> 使用 BLOG_SOURCE=directory 和 BLOG_DIR=/Users/acelee/workspace/blog 在本机构建 Astro
  -> DEPLOY_ENABLE_DIRECT=1 npm run deploy
```

以下记录仅用于手动操作旧服务端拉取部署脚本。

## 链路流程

```text
blog 推送
  -> ta 上的 Hermes cron 拉取 astro-blog 与 blog 内容源仓库
  -> ta 在本地构建 Astro
  -> rsync 将发布包同步到 /home/ubuntu/nginx-blog/html/
```

## 服务端文件

- `/usr/local/bin/astro-blog-pull-deploy`
- `/home/ubuntu/astro-blog-src/.env`
- `/etc/systemd/system/astro-blog-pull-deploy.service`
- `/home/ubuntu/.hermes/scripts/astro_blog_deploy_check.py`
- `/home/ubuntu/.hermes/cron/jobs.json`
- `/home/ubuntu/astro-blog-src`
- `/home/ubuntu/blog-src`
- `/home/ubuntu/nginx-blog/releases`
- `/home/ubuntu/nginx-blog/.deployed-source-state`
- `/home/ubuntu/nginx-blog/.deployed-posts.json`

ta 服务器通过 `ssh.github.com:443` 使用只读 GitHub 部署密钥访问两个私有仓库。GitHub Actions 已在这条部署链路中停用。

服务端环境变量以 `.env.example` 为准。生产环境 `.env` 应放在已检出的 Astro 项目旁边，即 `/home/ubuntu/astro-blog-src/.env`，不要放在 `/etc` 下。

## 运维操作

检查 Hermes cron 任务：

```bash
hermes cron list
hermes cron status
```

立即运行一次部署：

```bash
hermes cron run b8c998d627b1
hermes cron tick
```

读取日志：

```bash
journalctl -u astro-blog-pull-deploy.service -n 100 --no-pager
```

该服务具备幂等性。如果最新的 `astro-blog` 与 `blog` 提交已经部署，它只会运行健康检查。

Hermes cron 任务名为 `astro-blog deploy watcher`。它的预运行脚本使用 Hermes 的唤醒门约定：无变更运行会以 `{"wakeAgent": false}` 结束，因此 Hermes 会跳过 agent，也不会发送通知。

通知规则：

- 部署失败总是唤醒 Hermes，并发送飞书通知。
- 部署成功时，只有已部署文章的 slug 集合发生变化才会唤醒 Hermes。
- 仅源码变更、样式变更、脚本变更、以及既有文章内容编辑，在文章集合不变时会静默部署。
- 文章快照从生成后的 `dist/post/*/index.html` 输出读取，并持久化到 `/home/ubuntu/nginx-blog/.deployed-posts.json`。
- slug 变化会记录为一篇文章移除和一篇文章新增。
